# Apple WatchヘルスデータMCPパイプライン 要件定義

## 1. 背景

Apple Watchで記録されるヘルスケアデータ(アクティビティ、ワークアウト、睡眠等)を、Claude Codeから自然言語で分析できるようにしたい。データはHealth Auto Export(iPhoneアプリ、以下HAE)経由で取得する。

## 2. 前提条件・既知の課題

- HAEの「Sync to Mac」機能は、アプリ内部のiCloudストア向けの仕組みであり、外部からファイルとして直接読み取ることはできない。
- HAEはエネルギー量の内部/デフォルト単位が **kJ(キロジュール)** になっている場合があり、表示・変換設定を誤るとkcal表示に対してkJの数値がそのまま出る(約4.184倍)不具合が起こりうることを確認済み。アプリのUnit Preferencesを明示的にkcalへ設定することで解消することを確認済み。
- 過去に同期・エクスポートされたデータには、上記の単位バグが混入している可能性があるため、取り込み時に単位の正規化処理を必須とする。
- **[未検証・要確認]** 本設計はHAEのAutomations機能がPOSTリクエストにカスタムHTTPヘッダー(`Authorization: Bearer <secret>`)を付与できることを前提としている。この対応可否は実装着手前に実機で検証すること。非対応と判明した場合は、クエリパラメータ方式など代替の認証手段を検討する(基本設計.md §3参照)。

## 3. 目的

- Apple Watch/iPhoneで記録されたヘルスデータを継続的に取り込み、Claude Codeから自然言語で問い合わせ・トレンド分析できる状態にする。
- 分析のたびにiPhone側アプリを起動・前面表示しておく、あるいはMac側の常駐プロセスを維持する、といった運用負荷を可能な限り排除する。

## 4. 機能要件

| ID | 要件 |
|---|---|
| F-1 | iPhoneのHAEから定期的(目安: 1時間に1回程度。HAE側のAutomation設定に依存)に自動でヘルスデータを送信し、サーバー側で受信・保存できること |
| F-2 | 受信したデータのうち、エネルギー量などの単位が不正な場合は正規化(kJ→kcal変換など)した上で保存すること。未知の単位を受信した場合はエラーとして記録し、誤った値のまま保存しないこと |
| F-3 | Claude CodeからMCP(Model Context Protocol)経由で、保存済みデータに自然言語で問い合わせできること |
| F-4 | 期間を指定したメトリクスのサマリー(平均・合計・最大・最小)を取得できること |
| F-5 | ワークアウト・睡眠データを期間指定で取得できること |
| F-6 | 指標のトレンド(増減傾向)を算出できること |
| F-7 | 自宅Wi-Fi以外のネットワーク(外出先・4G/5Gなど)からでもiPhone側の自動送信が機能すること(→ パブリックなAPIエンドポイントを介するためネットワーク経路に依存しない。基本設計.md §1参照) |
| F-8 | 同一データが再送・重複送信された場合でも、重複してレコードが増えたり集計値が二重計上されたりしないこと(冪等性) |

## 5. 非機能要件

| ID | 要件 |
|---|---|
| N-1 | 個人利用規模(単一ユーザー)を前提とし、過剰な構成は避けること |
| N-2 | 常時稼働させるサーバー・デーモン・PCを必須としないこと(アイドル時のコストがかからない構成が望ましい) |
| N-3 | 月額コストは可能な限り無料利用枠内に収まることを目指すこと。無料枠を超過した場合に検知できること |
| N-4 | データ送受信のエンドポイントは認証必須とし、第三者が不正にデータを送信・取得できないようにすること。総当たり等の不正アクセス試行を軽減する仕組み(レート制限等)を備えること |
| N-5 | 将来的に問い合わせツール(MCPツール)を追加しやすい構成であること |
| N-6 | 保存したヘルスデータの保持期間: 当面は無期限保持とする。DynamoDBストレージコストが無視できない水準になった場合、削除・アーカイブ方針を別途検討する |
| N-7 | 障害時(iPhone側の送信失敗、Lambda処理エラー等)にデータ欠損が起きても、次回同期で自動的に補完される、または少なくともCloudWatch Logsから検知できること(常時監視・即時通知までは求めない) |

## 6. スコープ外

- 複数ユーザー対応(家族アカウント等)
- Apple Health以外のウェアラブル(Garmin、Whoopなど)との統合
- モバイルアプリ・Webダッシュボード等のUI構築
- リアルタイム(秒単位)でのデータ反映
- 高可用性(マルチリージョン、SLA保証等)の作り込み(個人利用のためN-1と整合)

## 7. 検討した代替案と選定理由

| 案 | 概要 | 不採用/保留理由 |
|---|---|---|
| HAE「Sync to Mac」をそのまま利用 | 既存運用 | アプリ内部ストアのみで、外部から直接ファイルアクセスできない |
| `health-auto-export-mcp-server`(TCPサーバー方式) | iPhone上のTCPサーバーへ直接クエリ | iOSのバックグラウンド実行制約により、分析中iPhoneアプリをForegroundで開き続ける必要がある |
| `apple-health-mcp`(alphonsekoh作、DuckDB) | LAN内のMacでingestデーモン+MCPサーバーを常駐 | 常時稼働プロセスが必要。LAN到達性が前提(Tailscale併用で緩和は可能) |
| `open-wearables`(the-momentum作) | 複数ウェアラブル対応のセルフホスト統合基盤+MCPサーバー | Docker常時稼働のバックエンド一式が必要で個人利用にはオーバースペック。MCPツールも現状4種類のみで発展途上 |
| **AWS API Gateway + Lambda + DynamoDB(サーバーレス)** | 取り込み・問い合わせの両方をLambdaで実装し、常駐プロセスを排除 | **採用**。N-2/N-3/F-7を満たせる |

## 8. 採用方針

AWS上にサーバーレス構成(API Gateway + Lambda + DynamoDB)を構築し、HAEのREST API自動化からのプッシュ受信と、Claude CodeからのMCP(HTTP transport)問い合わせの両方を、常駐プロセスなしで実現する。詳細は「基本設計.md」を参照。

## 9. 用語集

| 用語 | 説明 |
|---|---|
| HAE | Health Auto Export(iPhoneアプリ)。Apple Healthのデータをエクスポート・自動送信する |
| MCP | Model Context Protocol。LLMクライアント(Claude Code等)がツールを呼び出すためのプロトコル |
| pk / sk | DynamoDBのパーティションキー / ソートキー |
| GSI | DynamoDBのグローバルセカンダリインデックス |
| TTL | DynamoDBのTime To Live。指定時刻を過ぎたアイテムを自動削除する機能 |
